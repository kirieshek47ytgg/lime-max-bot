# Lime MAX Bot

Автоматизация уведомлений базы отдыха: клиент бронирует беседку в **Restoplace**,
а мы автоматически отправляем ему сообщение в мессенджере **MAX** через
самохостинг-шлюз **kapuchino-api** (личный аккаунт MAX, аналог green-api).

```
Restoplace (webhook) ──POST──▶ Lime MAX Bot (FastAPI) ──▶ kapuchino-api ──▶ MAX ──▶ клиент
```

Шлюз `kapuchino-api` держит привязанный по QR личный аккаунт MAX и принимает
REST `POST /api/v1/max/sendMessage` (Bearer-токен). Отправка идёт **по номеру
телефона гостя** (`chatId` вида `+79991234567`) — шлюз сам резолвит номер в
внутренний chatId MAX.

Помимо приёма webhook и отправки уведомлений есть полноценная **админ-панель
оператора** (`/admin`): диалоги, брони, шаблоны уведомлений, аналитика и управление
(см. ниже). Чего пока нет — обработки входящих сообщений **от гостей** (guest → бот):
их webhook принимается заглушкой и игнорируется, это следующий этап.

---

## Что отправляется

На события Restoplace (`reserve.created`, `reserve.deleted` — настраивается)
бот формирует и шлёт клиенту сообщение, например:

```
✅ Бронирование принято!

👤 Юлия
🏡 Беседка, Основной зал
📅 27.06.2026
🕐 13:00–14:00
👥 Гостей: 3
🔖 Бронь №101
```

В консоли при этом пишется опрятная сводка:

```
📩 reserve.created  (бронь #101)
   Гость:  Юлия, 79991112222
   Когда:  27.06.2026 13:00–14:00
   Объект: Беседка, Основной зал
   Гостей: 3
```

## Требования

- Python 3.11+
- Запущенный шлюз **kapuchino-api** с **привязанным аккаунтом MAX** (соседний
  проект): подними его одной командой `./start.ps1` (ядро + MAX-сайдкар) и привяжи
  аккаунт по QR в его панели. По умолчанию слушает `http://127.0.0.1:8080`.
- Доступный из интернета HTTPS-адрес (для приёма webhook). Для локального запуска —
  туннель `localtunnel` (см. раздел «Развёртывание через localtunnel»).
- Для туннеля: Node.js (18+) и пакет `localtunnel` (`npm i -g localtunnel`).

## Установка

```bash
python -m venv .venv
# Windows:
.venv\Scripts\activate
# Linux/Mac:
source .venv/bin/activate

pip install -r requirements.txt

# скопируйте шаблон конфигурации и заполните своими данными
copy .env.example .env      # Windows
# cp .env.example .env       # Linux/Mac
```

### Заполните `.env`

| Переменная | Описание |
|---|---|
| `GATEWAY_URL` | Адрес ядра шлюза kapuchino-api (по умолчанию `http://127.0.0.1:8080`) |
| `GATEWAY_TOKEN` | Bearer-токен REST API шлюза (= `API_TOKEN` из `.env` проекта kapuchino-api) |
| `GATEWAY_CHANNEL` | Канал шлюза: `max` (можно `whatsapp`/`telegram`, если привязаны) |
| `TEST_CHAT_ID` | Если задан — все уведомления уходят в один чат (`+79991234567` или числовой chatId). Пусто — слать гостю по его номеру |
| `WEBHOOK_PATH` | Путь эндпоинта (по умолчанию `/webhook/restoplace`) |
| `WEBHOOK_SECRET` | Необязательный секрет; если задан — Restoplace шлёт на `URL?token=...` |
| `ENABLED_NOTIFICATIONS` | Типы уведомлений через запятую: `created`, `updated`, `cancelled` |
| `ITEMS_FILE` | Файл-справочник `item_id → название` (по умолчанию `items.json`) |
| `DRY_RUN` | `true` — не отправлять в шлюз, только логировать (для тестов) |
| `SIGNATURE` | Необязательная подпись внизу сообщения |

> ⚠️ Пока `DRY_RUN=true`, **реальные сообщения не уходят** — удобно для проверки.
> Для боевого режима поставьте `DRY_RUN=false`, запустите шлюз kapuchino-api с
> привязанным MAX и заполните `GATEWAY_URL` / `GATEWAY_TOKEN`.

## Админ-панель (`/admin`)

Веб-кабинет оператора: переписка с гостями, брони, шаблоны уведомлений,
аналитика и управление. Открывается по адресу **http://localhost:8000/admin**.

- При первом запуске автоматически создаётся **владелец** панели с логином и
  паролем из `.env` (`ADMIN_USER` / `ADMIN_PASSWORD`, по умолчанию
  `admin` / `admin12345`). **Сразу смените пароль** во вкладке «Управление».
- Локально по http оставьте `COOKIE_SECURE=False`, иначе браузер не примет
  cookie сессии и вход не сработает.
- Данные хранятся в SQLite (`DB_PATH`, по умолчанию `bot_database.db`). Каждое
  событие из Restoplace пишется в панель (клиент, бронь, отправленное
  уведомление) — даже если отправка отключена.

| Вкладка | Что внутри |
|---|---|
| **Диалоги** | Вся переписка с гостями: отправленные уведомления и ручные ответы менеджера. Поиск, фильтры, отправка текста/файлов, правка и удаление сообщений, карточка гостя. |
| **Заказы** | Список броней со статусами (забронированы / завершены / отменены), смена статуса, карточка гостя, импорт/экспорт CSV, звук при новой брони. |
| **Бот** | Шаблоны сообщений для трёх событий (подтверждение / изменение / отмена) с плейсхолдерами `{имя}{объект}{дата}{время}{гостей}{номер}` и живым предпросмотром. Шаблоны хранятся в браузере. |
| **Аналитика** | Брони, гости, подтверждения/изменения/отмены, уведомления, доля доставленных; график за 14 дней, топ объектов, воронка уведомлений. |
| **Управление** | _(только владелец)_ Состояние систем (Restoplace / kapuchino-api / MAX), рубильник отправки и выбор событий, тестовый режим, журнал сбоёв, админы/роли, журнал входов, бэкап БД. |

Рубильник отправки, список отправляемых событий и тестовый режим в панели
**переопределяют** `ENABLED_NOTIFICATIONS` / `DRY_RUN` из `.env` (настройки
хранятся в БД и применяются на лету).

## Безопасность (чек-лист перед боем)

Когда панель открыта наружу через туннель — она доступна из интернета. Обязательно:

1. **Смените пароль владельца** во вкладке «Управление» (дефолт `admin/admin12345`
   годится только для первого входа).
2. Держите **`WEBHOOK_SECRET`** непустым — тогда чужой POST на вебхук отсекается (403).
3. **`GATEWAY_TOKEN`** и `.env` не коммитьте (уже в `.gitignore`).
4. По желанию `COOKIE_SECURE=True`, если панель открываете только по HTTPS-туннелю.

Что уже сделано на стороне кода:

- Пароли админов хранятся как **PBKDF2-хеши** (200k итераций), сравнение — постоянное по времени.
- **Антиперебор входа:** после 10 неудачных попыток с одного IP за 15 минут вход
  временно блокируется (429). Все попытки пишутся в журнал входов.
- **Сессии протухают** через 30 дней и подчищаются на сервере; cookie сессии — `HttpOnly`.
- Все эндпоинты панели (`/api/dashboard/*`) закрыты авторизацией, разделы
  «Управление» — дополнительно ролью `owner`.
- Загруженные из панели файлы сохраняются под безопасным именем (без обхода пути).

## Развёртывание через localtunnel

Restoplace шлёт webhook только на публичный HTTPS-адрес. Чтобы отдать наружу
локально запущенного бота, поднимаем туннель `localtunnel`.

```bash
# один раз ставим пакет (нужен Node.js)
npm install -g localtunnel
```

Запуск в два окна терминала:

```powershell
# окно 1 — сам бот на http://localhost:8000
./run.ps1

# окно 2 — публичный туннель к нему (стабильный адрес через -Subdomain)
./run-lt.ps1 -Subdomain lime-max
```

`run-lt.ps1` выведет публичный адрес вида `https://lime-max.loca.lt`. Полный
webhook для Restoplace:

```
https://lime-max.loca.lt/webhook/restoplace?token=<WEBHOOK_SECRET из .env>
```

> **Заметки по localtunnel**
> - При первом заходе **браузером** на `*.loca.lt` показывается страница-заглушка
>   localtunnel (нужно нажать «Continue»). На POST-запросы вебхука от Restoplace
>   она не влияет — они проходят напрямую.
> - Поддомен через `-Subdomain` не гарантирован (может быть занят) — но если
>   свободен, адрес не меняется между перезапусками, и webhook в Restoplace
>   не приходится переписывать.
> - Туннель по HTTPS — можно выставить `COOKIE_SECURE=True` в `.env`, если панель
>   вы открываете только через него (а не по http://localhost).

## Подключение Restoplace

1. Запустите бота и туннель (см. раздел выше) — либо разместите бота на сервере с HTTPS.
2. В Restoplace: **Общие настройки → Уведомления SMS/Мессенджер**.
3. Включите тумблер **Webhook** и вставьте адрес:
   `https://ВАШ_ДОМЕН/webhook/restoplace`
   (если задан `WEBHOOK_SECRET` — добавьте `?token=ВАШ_СЕКРЕТ`).
4. Сделайте тестовую бронь и проверьте лог бота.

## Доставка в MAX: важно

Сообщение придёт, **только если у номера гостя установлен и зарегистрирован MAX**.
MAX-сайдкар шлюза резолвит номер (`+79991234567`) в внутренний chatId; если у
номера нет аккаунта MAX (или он скрыт настройками приватности), шлюз вернёт
ошибку `404 — У номера ... нет аккаунта MAX`, бот запишет её в журнал сбоёв и
ответит Restoplace `502` (чтобы тот мог повторить доставку).

Диагностика шлюза (подставьте свой `GATEWAY_TOKEN`):

```bash
# Состояние всех каналов (для MAX state должно быть "authorized")
curl -H "Authorization: Bearer <GATEWAY_TOKEN>" http://127.0.0.1:8080/api/v1/status
# Пробная отправка по номеру
curl -X POST http://127.0.0.1:8080/api/v1/max/sendMessage \
  -H "Authorization: Bearer <GATEWAY_TOKEN>" -H "Content-Type: application/json" \
  -d '{"chatId":"+79991234567","message":"тест"}'
```

> Примечание: `POST /api/webhook` — это шлюз шлёт нам входящие сообщения (если в
> kapuchino-api задан `WEBHOOK_URL` на наш адрес). Пока они не обрабатываются:
> эндпоинт-заглушка отвечает `200`. Успешные запросы в консоль не пишутся —
> лог-фильтр оставляет только ошибки (4xx/5xx), чтобы не засорять вывод поллингом.

## Реальный формат webhook Restoplace

⚠️ Фактический формat отличается от справки Restoplace. По факту приходит
events-формат (подтверждено на боевых данных):

| Поле | Значение |
|---|---|
| `event` | `reserve.created` / `reserve.updated` / `reserve.deleted` |
| `reserve_id` | номер брони |
| `name` | имя гостя |
| `phone` | телефон гостя |
| `count` | количество гостей |
| `time_from` / `time_to` | период брони `YYYY-MM-DD HH:MM:SS` |
| `item_type` | `alcove` (беседка), `table` (столик), `waitlist` (лист ожидания) |
| `item_id` | внутренний id объекта (НЕ человекочитаемый номер беседки) |
| `floor_name` | название зала/зоны |
| `status` | `2` — активна, `6` — отменена |
| `cancel_reason`, `time_cancel`, `userid_cancel` | заполняются при отмене |
| `success`, `source` | служебные поля |

Маппинг описан в [app/models.py](app/models.py) и [app/messages.py](app/messages.py).
Полный сырой payload пишется в лог на уровне `DEBUG` (`LOG_LEVEL=DEBUG`).

### Уведомления об изменении брони (updated)

`reserve.updated` шлётся клиенту **только если изменилось значимое для него поле**.
Webhook присылает снимок целиком (а не дельту), поэтому бот хранит предыдущий снимок
каждой брони в `STATE_FILE` ([state/reservations.json]) и сравнивает.

**Шлём при изменении:** даты/времени (`time_from`/`time_to`), беседки (`item_id`),
типа объекта (`item_type`), зоны (`floor_name`), числа гостей (`count`).
**Не шлём** при правке внутренних полей: заметок (`text`), тегов (`tags`), статуса,
депозита/оплаты, служебных полей. Список настраивается в `CLIENT_FIELDS`
([app/messages.py](app/messages.py)).

Если базового снимка нет (бот был выключен в момент создания брони) — правка не
шлётся, снимок сохраняется как базовый для последующих сравнений.

### Отмена брони

⚠️ Отмена приходит **не отдельным событием**, а как `reserve.updated` со
`status: 6` (и заполненными `cancel_reason` / `time_cancel`). Бот определяет
смысл события (`created` / `updated` / `cancelled`) по этим полям, а не по имени
события — см. `logical_event()` в [app/messages.py](app/messages.py).

### Номер беседки (справочник items.json)

В webhook **нет человекочитаемого номера беседки** — только внутренний `item_id`
(например `789368`) и зона `floor_name`. Чтобы показывать гостю «Беседка №2»,
используется справочник [items.json](items.json):

```json
{ "789368": "Беседка №2" }
```

Чтобы узнать `item_id` нужной беседки — сделайте на неё тестовую бронь и
посмотрите в логе строку `Объект: ... [item_id=... — добавьте в items.json]`,
затем впишите пару `"item_id": "Название"` в `items.json` и перезапустите бота.
Если объекта нет в справочнике — выводится тип и зона («Беседка, Основной зал»).

## Структура проекта

```
app/
  main.py          — FastAPI: приём webhook, запись событий в БД, монтаж панели
  config.py        — настройки из .env (вкл. параметры панели)
  models.py        — модель payload Restoplace
  messages.py      — смысл события + сборка текста + строка для лога
  gateway.py       — клиент kapuchino-api: отправка текста/файла, состояние канала
  phone.py         — нормализация телефона -> chatId (+79991234567)
  items.py         — справочник item_id -> название объекта
  state.py         — снимки броней (диф значимых правок)
  db.py            — SQLite-хранилище панели (клиенты, брони, сообщения, админы)
  auth.py          — вход/сессии панели (PBKDF2, cookie)
  dashboard_api.py — REST API панели (/api/dashboard/*)
  web.py           — отдача страницы панели (/admin)
  static/panel.html — вся веб-панель (вёрстка + стили + JS в одном файле)
  notify.py        — дублирование важных событий владельцу в Telegram
items.json         — данные справочника беседок/столов
run.ps1            — запуск бота на Windows (создаёт .venv, ставит зависимости, поднимает uvicorn)
run-lt.ps1         — публичный HTTPS-туннель к боту через localtunnel (для webhook)
tests/             — pytest + примеры payload
```
